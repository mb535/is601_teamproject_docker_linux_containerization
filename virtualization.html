Virtualization!


We've come a long way from giant servers and computers that filled rooms and even warehouses. Proving Moore's Law, as more and more transistors were able to be put on a chip, the computers grew more and more powerful. Due to the popularity of the x86 architecture, majority of servers used processors based on this architecture as their central processing units (CPUs). From file servers to database, web, mail and all other servers, as the Internet grew this explosive growth in servers presented challenges. From the physical footprint required for these servers, to the need for acquiring new hardware, to being utilized for only a fraction of their capacity, Virtualization rose as an solution.


Virtualization relies on software to simulate hardware functionality. From servers to disk to ports, virtualizing allows IT organizations to run more than one virtual system, regardless of the OS or applications, on one physical server. This allows for greater efficiencies and greater economies of scale. Where one would have had to, for example, deploy 4 servers (web, mail, file, domain controller, to establish an Internet presence for a small company, we can now do it with one physical server hosting all of them that are allocated resources per their use and function.


While Virtualization has brought a revolution, there are still several issues that persist with their use. Each virtual machine is still a full-fledged computer that must be managed as a standalone computer with regular OS and security patches. In a swimlane based application development architecture, the code must move from dev to testing to production to essentially a different computer each time, each one with it's own idiosyncrasies based on when and how it was last updated.

Containers attempt to solve these concerns. Containers are typically small units consisting an application and all it's dependencies. They typically do not contain the host OS or any of it's components, such as device drivers, display drivers and such. They contain everything that is needed to run the code and achieve the intended functionality. The containers are deployed on top of engines which provide the interface to the host OS and provide the application in the container all the services it needs.

If the container engine is universal, meaning it can run on multiple OSs, the container itself will be able to take advantage of that and be available across different hosts.

A big advantage of Containers is it isolates the software from it's environments, and ensures it works uniformly across the environment. In the swimlane example discussed earlier, the same container would flow along different environments, installed "on" the container engine. This ensures the code functions the same on each instance.
